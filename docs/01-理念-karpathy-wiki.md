# 设计理念：借鉴 Karpathy 的 LLM Wiki

## 问题从哪来

求职准备时，经历散落在各处。每次写简历或准备面试，都要重新翻一遍资料、重新喂给 AI。这有三个痛点：

1. **重复劳动**：同一个项目，写简历翻一遍、写逐字稿翻一遍、换个岗位再翻一遍
2. **又贵又慢**：每次把几千字的原始资料喂给 AI，token 成本高、还容易丢细节
3. **数据对不齐**：简历里写 CAC 22，逐字稿里记成 23，因为各写各的

## Karpathy 的思路

Andrej Karpathy 在 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 里提出一个知识库模式，核心是：

> **不要在每次查询时才从原始文档里检索，而是让 LLM 增量地构建并维护一个持久的 wiki**——知识被"沉淀"下来，而不是每次重新推导。

他的三层结构：
1. **Raw sources**（原始文档，不可变）
2. **Wiki layer**（LLM 生成的、带交叉引用的 markdown 笔记）
3. **Schema**（指导 LLM 行为的配置）

省 token 的关键：先查**索引**定位相关页面，只读相关的几页，而不是把整个库塞进上下文。

## 本 Agent 怎么落地

| Karpathy | 本 Agent | 说明 |
|---|---|---|
| Raw sources | `materials/` | 你粘贴的原始经历 dump，不动 |
| Wiki layer | `经历库/` | AI 整理成面试就绪的完善档案 |
| Index | `wiki/index.md` | 轻量定位表，生成时先看它 |
| Schema | `SKILL.md` | 指导 AI 怎么整理、怎么生成 |

### 工作流

```
① 你把原始经历粘进 materials/
② 对 AI 说「整理进经历库」
   → AI 读原始素材，按提炼标准（量化数据全保留、三段式、why+how、人设亮点）
     整理成 经历库/ 里的一篇完善档案，并更新索引
③ 你说「写简历」或「写某项目逐字稿」
   → AI 先读 wiki/index.md 定位 → 只读相关经历库档案 + 模板 → 生成到 output/
```

### 为什么这样省 token 又保质量

- **省 token**：生成时不读全部原始资料，只读索引 + 相关的 1-2 篇精炼档案。你有 10 个项目时，写其中一个只取那一个。
- **保质量**：经历库是「整理过一次」的结果，量化数据齐、结构清晰，比每次临时从原始资料里抓更稳定。
- **数据对齐**：简历、逐字稿、JD 定制版都从同一个经历库出，数字永远一致。

## 为什么不用 RAG：量级决定方案

很多人一提"知识库 + AI"就想上 RAG（向量数据库 + 检索）。但 RAG 不是默认答案，**该用什么取决于知识库的量级**：

| 知识库量级 | 推荐方案 | 原因 |
|---|---|---|
| **< 50k–100k tokens**（约 150–200 页） | **LLM Wiki（上下文模式）** | 简单、可靠、零维护成本 |
| 数百万 tokens 以上 | RAG（检索） | 塞不进上下文，只能靠检索扩展 |
| 介于两者之间 / 生产系统 | 混合：核心稳定知识进上下文，海量动态数据走 RAG | 兼顾 |

**一份个人的经历库，整理后通常只有几千到一两万 token，远低于 10 万 token 的界限**——所以这个场景下，上下文模式（LLM Wiki）完胜 RAG：

- **检索可靠性 100%**：所有信息都在 AI「视线」内，不会出现 RAG「没匹配到相关片段」或「分块切断语义」的问题
- **系统复杂度极低**：只需结构良好的 Markdown 文件，**不需要向量数据库、嵌入模型、分块逻辑**——从数周开发缩短到几小时准备
- **推理更精准**：AI 能对全局信息直接推理（多跳推理、跨经历总结），而不只是基于检索到的几个孤立片段

现代主流模型的上下文窗口已达 **200k 甚至 100 万 tokens**，进一步抬高了上下文模式的适用上限——个人经历库这种量级，根本不需要 RAG。

## 那本项目的「索引」算 RAG 吗？

不算。本项目**整体仍是上下文模式**——经历库小到完全能放进上下文。`wiki/index.md` 这个索引只是一个**轻量优化**：当经历多起来时，先看索引、只读相关的几篇，省掉不必要的 token；但它读的是完整的 Markdown 档案，不做向量匹配、不做分块。即便把整个经历库全读进去也放得下，索引只是让它更省而已。

所以本质区别：
- **RAG**：每次查询去原始文档里**向量检索**碎片，再拼接
- **本模式**：先把知识**沉淀**成结构化档案，查询时在档案层**按文件名定位**整篇读取

对求职这种**反复使用同一批经历、重视叙事质量**的场景，沉淀式更合适——面试逐字稿要的是经过提炼的判断力叙事，而不是原始片段的拼接。

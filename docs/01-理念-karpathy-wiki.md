# 设计理念：借鉴 Karpathy 的 LLM Wiki

## 问题从哪来

求职准备时，经历散落在各处。每次写简历或准备面试，都要重新翻一遍资料、重新喂给 AI。这有三个痛点：

1. **重复劳动**：同一个项目，写简历翻一遍、写逐字稿翻一遍、换个岗位再翻一遍
2. **又贵又慢**：每次把几千字的原始资料喂给 AI，token 成本高、还容易丢细节
3. **数据对不齐**：简历里写 CAC 22，逐字稿里记成 23，因为各写各的

## Karpathy 的思路

Andrej Karpathy 在 [LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) 里提出一个知识库模式，核心是：

> **不要在每次查询时才从原始文档里检索，而是让 LLM 增量地构建并维护一个持久的 wiki**——知识被"沉淀"下来，而不是每次重新推导。

他的三层结构：
1. **Raw sources**（原始文档，不可变）
2. **Wiki layer**（LLM 生成的、带交叉引用的 markdown 笔记）
3. **Schema**（指导 LLM 行为的配置）

省 token 的关键：先查**索引**定位相关页面，只读相关的几页，而不是把整个库塞进上下文。

## 本 Agent 怎么落地

| Karpathy | 本 Agent | 说明 |
|---|---|---|
| Raw sources | `materials/` | 你粘贴的原始经历 dump，不动 |
| Wiki layer | `经历库/` | AI 整理成面试就绪的完善档案 |
| Index | `wiki/index.md` | 轻量定位表，生成时先看它 |
| Schema | `SKILL.md` | 指导 AI 怎么整理、怎么生成 |

### 工作流

```
① 你把原始经历粘进 materials/
② 对 AI 说「整理进经历库」
   → AI 读原始素材，按提炼标准（量化数据全保留、三段式、why+how、人设亮点）
     整理成 经历库/ 里的一篇完善档案，并更新索引
③ 你说「写简历」或「写某项目逐字稿」
   → AI 先读 wiki/index.md 定位 → 只读相关经历库档案 + 模板 → 生成到 output/
```

### 为什么这样省 token 又保质量

- **省 token**：生成时不读全部原始资料，只读索引 + 相关的 1-2 篇精炼档案。你有 10 个项目时，写其中一个只取那一个。
- **保质量**：经历库是「整理过一次」的结果，量化数据齐、结构清晰，比每次临时从原始资料里抓更稳定。
- **数据对齐**：简历、逐字稿、JD 定制版都从同一个经历库出，数字永远一致。

## 与纯 RAG 的区别

RAG 是「每次查询时去原始文档里检索片段」。本模式是「先把知识沉淀成结构化档案，查询时在档案层检索」。对求职这种**反复使用同一批经历、且重视叙事质量**的场景，沉淀式更合适——因为面试逐字稿需要的是经过提炼的判断力叙事，而不是原始片段的拼接。
